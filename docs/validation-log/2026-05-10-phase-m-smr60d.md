## Diagnosis — why so few fires

Per the `b2b_pup` port agent's own report (verbatim quote in
`bible-input/extract-b2b-pup.yaml` comment header):

> "S-plots whose fire conditions include only stateful inputs will
> silently return False until Phase 2 stateful engine is added.
> S1 (B2B PUP/PPD), S2 (FAUNA), S3/S4 (DISP), S5 (RVOL), S8 (UC), S11
> (FVG+L1), S19/S20 (UC2/FMU) can all fire when non-stateful inputs are
> met. S6/S12/S13/S14/S15/S16/S17/S18 have PBJ/UU/Napalm/TNT gates that
> are stubs."

Same applies to most of the heavy stateful packs (`hvd_pbj_ppd`,
`fauna_shifu`, `tnt_od`, `swings`, `heavy_combo`, `anish_stack`). The
runner instantiates **fresh state objects per detect() call** instead
of threading them across the full bar history. That makes every state
machine reset on each call, so:
- PBJ engine has no level-bank history → all `PBJ_BULL` / `PBJ_BEAR`
  fires return False.
- Supertrend ratchet has no level history → `PB_*` rarely fires.
- TB/Foster state machines never accumulate the required N bars →
  composites that depend on them stay silent.
- TNT zone bookkeeping (Napalm, Charge, CONT) is stateful → silent.
- VOB engine actually works because its `state` is threaded internally
  (per-call object retains arrays).
- `primitives` works partly because HEV/HV-ladder/disp roots are
  stateless lookbacks.

## Phase 2 work required (not blocking Stage 3 verification)

The runner needs a state-threading pass: instantiate one state object
per (ticker, TF, pack), call `pack.detect_bar(state, bars, bar)` per
bar so state machines accumulate properly. The packs already have
state classes (`PrimitivesState`, `UltraComboState`, `FaunaShifuState`,
`E3F2ClusterState`, etc.) — the runner just needs to thread them.

## What Stage 3 verification uses INSTEAD

Anish's actual TradingView chart is the runtime ground truth. The
Path A logger Pines (`path-a-loggers/versions/LOGGER_*.pine`) emit
one `label.new()` per fire — readable in bulk via
`data_get_pine_labels(study_filter="LOGGER ...")`. That gives the
real fire history for every composite, sidestepping the Python state
limitation entirely.

For Stage 4 (composite firing validation), use:
1. Pine logger labels → real fire history per composite.
2. The bible YAML composition spec → what SHOULD fire.
3. Reconcile.

Phase M's role is supportive (cross-checking against Pine), not
primary. Until the realtime-indicators state-threading lands, Pine is
the authoritative reference and the loggers are the bulk-readout.
